# FRAMEWORK DECISION MATRIX

## Purpose

Record the structured comparison of candidate agent frameworks and document
the selection decision with sufficient rigour to survive a hostile architecture review.

## Required Headings

1. Evaluation criteria table (criterion, weight, rationale for weight)
2. Candidate comparison table (framework × criterion → score + evidence note)
3. Winner and weighted totals
4. Rejected alternatives — for each: strongest argument for and specific reason rejected
5. Decision rights — who can request a framework change post-Day 3 and what evidence is required

## Guiding Questions

- Which criterion is hardest to evaluate objectively, and how did you handle that uncertainty?
- Which rejected alternative had the strongest single argument in its favour?
- If the winning framework reaches an API limit or deprecation at Day 10, what is the migration blast radius?
- Who would challenge the weighting in a hostile review, and on what grounds?

## Acceptance Criteria

- Minimum 4 evaluation criteria with weights (weights must sum to 100)
- Minimum 3 frameworks compared (including the one selected)
- Evidence note per cell is specific (not just "good" or "better")
- Rejected-alternatives section has one sentence for the strongest pro-argument per alternative
- Decision rights section names a role and names the evidence required (not just "approval needed")
