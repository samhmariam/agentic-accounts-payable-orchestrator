# GATE EXCEPTION POLICY

## Purpose

Define the conditions under which a production release gate may be waived,
the authority required, and the obligations that attach to any exception granted.

## Required Headings

1. Gate exception classification — which gates can never be excepted, which require CAB review, which may be waived with evidence
2. Exception request format — required fields when requesting a gate exception
3. Authority matrix — who can approve each exception class, and whether any gate requires co-signature
4. Exception obligations — what the requesting team must do after an exception is granted
5. Exception expiry — how long an exception is valid and what happens at expiry

## Guiding Questions

- Which gate would an executive be most likely to pressure you to waive for business urgency?
- What is the blast radius of approving an exception on the safety gate?
- How many consecutive exceptions in a release cycle should trigger a mandatory gate review?
- What evidence would an auditor require to verify that an exception was legitimately approved?

## Acceptance Criteria

- No-exception gates are explicitly listed (not implied by omission)
- Exception request format has required fields (not narrative)
- Authority matrix names roles, distinguishes single vs co-signature, and covers out-of-hours
- Exception obligations include a post-deployment verification step
- Exception expiry policy is a specific duration or event — not "until no longer needed"
